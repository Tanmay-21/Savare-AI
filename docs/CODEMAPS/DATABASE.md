# Codemap: Supabase PostgreSQL Database

## Connection

**Browser:** `src/supabase.ts` creates a Supabase client with `VITE_SUPABASE_ANON_KEY`. Used only for Auth (session management). Never used for direct data queries.

**Server:** `api/lib/supabase.ts` creates a Supabase admin client with `SUPABASE_SERVICE_ROLE_KEY`. Used by all API handlers. Bypasses Row Level Security (RLS) — access control is enforced manually via `user_id` filters in every query.

## Naming Convention

Database columns are `snake_case`. The `apiFetch()` utility in `src/lib/api.ts` automatically converts:
- Request bodies: `camelCase` → `snake_case` before sending
- Responses: `snake_case` → `camelCase` before returning to components

So `billing_party_name` in Postgres arrives as `billingPartyName` in React components.

## RPC Functions

### `get_next_counter(p_user_id, p_type, p_fiscal_year) → integer`
Atomically increments and returns the next sequence number for orders or trips. Uses `INSERT ... ON CONFLICT DO UPDATE` to avoid race conditions.

### `get_next_lr_number(p_user_id, p_fiscal_year) → integer`
Same pattern, dedicated to LR sequences.

**Gap tolerance:** Both RPCs increment the counter before the subsequent insert. If the insert fails (validation, network), the sequence number is consumed and a gap appears. This is intentional — gaps are acceptable and preferable to complex rollback logic.

## Denormalization Pattern

Vehicle numbers (`vehicle_number`) and driver names (`driver_name`) are copied into `shipments` and `expenses` at write time. This avoids joins on reads. **Existing records are not back-filled** if a vehicle plate or driver name changes later — they retain the value at the time of assignment.

## Fiscal Year

The Indian fiscal year runs April–March. Format: `YY-YY` (e.g. `25-26` for FY 2025-26). All sequence numbers reset per fiscal year per user. Logic is in `src/lib/fiscalYear.ts`.
